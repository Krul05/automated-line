# Документация веб-сервиса Montrac

## Назначение

Сервис управляет линией Montrac через IRM-станции, подключенные к Moxa Virtual COM ports. Он принимает события присутствия тележек от станций, ведет карту занятых станций и перегонов, отправляет команду `START FORWARD` в нужный COM-порт и предоставляет веб-интерфейс для оператора.

Основные возможности:

- запуск тележки с конкретной станции кнопкой `Запустить`;
- автоматические режимы работы с задержкой на каждой станции;
- редактирование списка станций через веб-интерфейс;
- редактирование режимов работы через веб-интерфейс;
- просмотр состояния станций, перегонов и журнала событий;
- mock-режим для проверки без реальных COM-портов.

## Состав проекта

| Файл | Назначение |
| --- | --- |
| `main.py` | Точка запуска сервиса, аргументы командной строки, старт контроллера и HTTP-сервера. |
| `montrac/controller.py` | Основная логика линии: станции, перегоны, режимы, отправка команд, сохранение конфигурации. |
| `montrac/protocol.py` | Формирование и разбор IRM-кадров. |
| `montrac/web.py` | HTTP API и встроенный веб-интерфейс. |
| `config.json` | Рабочая конфигурация линии. Создается при первом запуске, если отсутствует. |
| `config.example.json` | Базовый пример конфигурации. |
| `requirements.txt` | Python-зависимости. |
| `tests/` | Автотесты логики, протокола и веб-страницы. |

## Установка

Установите зависимости:

```powershell
pip install -r requirements.txt
```

Для работы с реальными портами используется пакет `pyserial`. На Windows COM-порты должны быть заранее созданы через Moxa NPort / Moxa Virtual COM Driver и быть видимыми в системе как `COM1`, `COM2` и так далее.

## Запуск

Обычный запуск:

```powershell
python main.py --config config.json --host 0.0.0.0 --port 8080
```

Открыть интерфейс:

```text
http://localhost:8080
```

Запуск без реального оборудования:

```powershell
python main.py --mock --port 8080
```

Аргументы запуска:

| Аргумент | По умолчанию | Описание |
| --- | --- | --- |
| `--config` | `config.json` | Путь к рабочему JSON-конфигу. Относительный путь считается от папки проекта. |
| `--host` | `0.0.0.0` | Адрес, на котором слушает HTTP-сервер. |
| `--port` | `8080` | HTTP-порт сервиса. |
| `--mock` | выключен | Не открывать реальные COM-порты, использовать mock-порты. |

Если `config.json` отсутствует, сервис создает его из `config.example.json`. Если пример тоже отсутствует, используется встроенная конфигурация COM1-COM6.

## Конфигурация

Пример структуры:

```json
{
  "hold_seconds": 3,
  "loop": true,
  "departure_grace_seconds": 1.5,
  "scheduler_interval_seconds": 0.1,
  "event_limit": 200,
  "stations": [
    { "index": 1, "name": "Station 1", "port": "COM1", "baudrate": 9600, "group": 1 },
    { "index": 2, "name": "Station 2", "port": "COM2", "baudrate": 9600, "group": 1 }
  ],
  "modes": [
    {
      "id": "stop_all",
      "name": "Все станции · 3 с",
      "stationDelays": {
        "1": 3,
        "2": 3
      }
    }
  ]
}
```

Поля верхнего уровня:

| Поле | Описание |
| --- | --- |
| `hold_seconds` | Базовая задержка в секундах для стандартных режимов. |
| `loop` | Если `true`, последняя станция отправляет тележку на первую. Если `false`, последняя станция не имеет следующей. |
| `departure_grace_seconds` | Время после отправки команды, в течение которого повторное сообщение от той же станции игнорируется как хвостовой сигнал отправления. |
| `scheduler_interval_seconds` | Интервал проверки автоматического режима. |
| `event_limit` | Максимальное количество событий в журнале контроллера. |
| `stations` | Список станций линии. |
| `modes` | Список режимов работы. |

Поля станции:

| Поле | Описание |
| --- | --- |
| `index` | Номер станции в маршруте. Используется для порядка движения. |
| `name` | Название в интерфейсе. |
| `port` | COM-порт Moxa, например `COM11`. |
| `baudrate` | Скорость порта, обычно `9600`. |
| `group` | IRM-группа для формирования команды. |

Поля режима:

| Поле | Описание |
| --- | --- |
| `id` | Технический идентификатор режима для API. |
| `name` | Название кнопки в интерфейсе. |
| `stationDelays` | Задержки по станциям. Ключ - номер станции, значение - секунды ожидания. |

Задержка `0` означает: отпустить тележку сразу, когда это разрешено текущей картой перегонов.

## Логика движения

Сервис строит маршрут по возрастанию `station.index`.

Если `loop: true`, маршрут замкнут:

```text
Station 1 -> Station 2 -> ... -> Station N -> Station 1
```

Если `loop: false`, последняя станция не отправляет тележку дальше.

При получении сообщения присутствия от IRM-станции сервис:

1. отмечает станцию занятой;
2. записывает ID тележки;
3. удаляет этот ID из любого перегона, где он ранее числился;
4. обновляет время ожидания на станции.

При запуске тележки со станции сервис:

1. проверяет, что станция существует;
2. проверяет, что на станции есть тележка;
3. определяет следующую станцию по маршруту;
4. проверяет, что в перегоне до следующей станции сейчас `0` тележек;
5. отправляет `START FORWARD` в COM-порт текущей станции;
6. освобождает текущую станцию в карте сервиса;
7. добавляет ID тележки в соответствующий перегон.

Текущее правило безопасности перегона: тележка отправляется только если в перегоне между текущей и следующей станцией сейчас `0` тележек. Если там уже есть хотя бы одна тележка, запуск блокируется.

Занятость следующей станции не используется как отдельная блокировка. Ограничение задается именно состоянием перегона.

## Режимы работы

Режим задает задержку на каждой станции. После выбора режима планировщик регулярно проверяет занятые станции и отправляет тележку, когда:

- задержка для этой станции истекла;
- следующий перегон свободен по правилу `0` тележек;
- COM-порт станции доступен.

Режим можно остановить кнопкой `Остановить режим`. Это отключает автоматические отправки, но не останавливает уже отправленную тележку.

Режимы можно создавать, удалять и редактировать в веб-интерфейсе в блоке `Режимы работы`.

## Веб-интерфейс

Главная страница доступна по `/`.

Основные блоки:

- верхние кнопки режимов - создаются из `modes`;
- `Станции` - состояние каждой станции и кнопка `Запустить`;
- `Перегоны` - количество тележек в каждом перегоне и их ID;
- `Журнал` - последние события контроллера;
- `Режимы работы` - редактор режимов;
- `Конфигурация` - редактор станций.

Редактировать станции можно только когда линия пустая: нет занятых станций и нет тележек в перегонах. Это нужно, чтобы сервис не потерял текущую карту движения.

## HTTP API

Подробные схемы endpoints, ответы, ошибки и примеры интеграции вынесены в отдельный документ: [api.md](api.md).

Все ответы API возвращаются в JSON.

### GET `/health`

Проверка, что HTTP-сервер жив.

Ответ:

```json
{ "ok": true }
```

### GET `/api/state`

Возвращает текущее состояние линии.

Основные поля ответа:

| Поле | Описание |
| --- | --- |
| `mode` | ID активного режима или `idle`. |
| `activeModeId` | ID активного режима или `null`. |
| `activeModeName` | Название активного режима или `null`. |
| `stations` | Список станций и их состояние. |
| `segments` | Список перегонов, количество тележек и ID. |
| `modes` | Список доступных режимов. |
| `events` | Последние события журнала. |

Пример запроса:

```powershell
curl http://localhost:8080/api/state
```

### POST `/api/actions/select_mode`

Выбрать режим работы.

Тело:

```json
{
  "modeId": "stop_all"
}
```

Остановить режим:

```json
{
  "modeId": null
}
```

Пример:

```powershell
curl -X POST http://localhost:8080/api/actions/select_mode `
  -H "Content-Type: application/json" `
  -d "{\"modeId\":\"stop_all\"}"
```

### POST `/api/actions/release_station`

Запустить тележку с конкретной станции.

Тело:

```json
{
  "station": 1
}
```

Если следующий перегон занят хотя бы одной тележкой, запуск будет отклонен.

Пример:

```powershell
curl -X POST http://localhost:8080/api/actions/release_station `
  -H "Content-Type: application/json" `
  -d "{\"station\":1}"
```

### POST `/api/config/stations`

Сохранить список станций. Доступно только когда линия пустая.

Тело:

```json
{
  "stations": [
    { "index": 1, "name": "Station 1", "port": "COM1" },
    { "index": 2, "name": "Station 2", "port": "COM2" }
  ]
}
```

### POST `/api/config/modes`

Сохранить список режимов.

Тело:

```json
{
  "modes": [
    {
      "id": "stop_all",
      "name": "Все станции · 3 с",
      "stationDelays": {
        "1": 3,
        "2": 3
      }
    }
  ]
}
```

### POST `/api/mock/presence`

Используется только для проверки в mock-режиме. Имитирует приход тележки на станцию.

Тело:

```json
{
  "station": 1,
  "shuttleId": 101,
  "group": 1
}
```

## IRM-протокол

Сервис использует 7-байтовые кадры:

```text
01 group id_high id_low command checksum 03
```

Контрольная сумма:

```text
group XOR id_high XOR id_low XOR command
```

Используемые команды:

| Код | Назначение |
| --- | --- |
| `83` | Присутствие тележки, сообщение приходит от IRM. |
| `51` | `START FORWARD`, команда отправляется сервисом в IRM. |

## Безопасность доступа

В текущей реализации HTTP API не имеет встроенной авторизации. Если запустить сервис с `--host 0.0.0.0`, команды управления смогут отправлять все устройства, у которых есть сетевой доступ к порту сервиса.

Минимальные меры защиты:

1. Размещать сервис только в закрытой локальной сети линии.
2. На Windows Firewall разрешить порт сервиса только с IP операторского ПК и доверенных устройств.
3. Не публиковать порт сервиса в интернет.
4. Для интеграции с роботом-манипулятором использовать отдельный endpoint и API-ключ.
5. Для удаленного доступа использовать VPN или reverse proxy с авторизацией.

Аппаратные средства безопасности линии, аварийный стоп, концевики и штатные блокировки оборудования должны оставаться главным уровнем защиты. HTTP-сервис не должен быть единственным защитным контуром.

## Проверка и диагностика

Запуск тестов:

```powershell
python -m unittest discover -v
```

Проверка синтаксиса Python:

```powershell
python -m compileall main.py montrac tests
```

Частые ситуации:

| Симптом | Что проверить |
| --- | --- |
| Станция отображается как отключенная | Номер COM-порта, Moxa Virtual COM Driver, доступность порта, занятость порта другой программой. |
| Кнопка `Запустить` неактивна | Есть ли тележка на станции, есть ли следующая станция, свободен ли следующий перегон. |
| Режим выбран, но тележки не уходят | Задержки режима, наличие тележек на станциях, занятость перегонов, ошибки COM-портов. |
| Конфигурация станций не сохраняется | Линия должна быть пустой: нет занятых станций и перегонов. |
| При первом запуске нет нужных станций | Проверьте путь `--config`; относительный путь считается от папки проекта. |

## Рекомендуемый порядок эксплуатации

1. Запустить Moxa Virtual COM Driver и убедиться, что COM-порты созданы.
2. Проверить `config.json`: станции, порты, `loop`, режимы.
3. Запустить сервис.
4. Открыть веб-интерфейс.
5. Проверить, что станции подключены.
6. Выбрать режим работы или запускать станции вручную кнопками `Запустить`.
7. При изменении станций сначала освободить линию, затем сохранить новую конфигурацию.
